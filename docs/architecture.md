系统由四个部分构成，形成以下依赖关系：

```
接入平面（读/写外部世界）
    ↓
流程引擎（维护实例状态和节点生命周期）
    ↓
决策机构
    ├── Agents（动作决策 + 评审提交 + 知识沉淀）
    └── 人工评审（批准/驳回，实际执行修改）
    ↓
知识库（接收 Agent 沉淀的内容）
```

### 3.1 接入平面

**职责**：连接外部世界与系统内部，双向通道。

- **读入方向**（事件标准化）：把飞书消息、GitHub webhook、定时任务等各种信息源的原始数据转化为系统内部的标准事件格式，流程引擎只消费标准事件，不关心数据来自哪里。
- **写出方向**（动作路由）：把系统内部的决策（Agent 建议、人的批准）路由到外部系统，如飞书通知、GitHub 打标签等。

**设计原则**：

- **标准事件格式**：新增信息源只需写一个新的 adapter，不需要改流程引擎（参考 Paperclip 的 adapter 模式，但应用于信息源而非 Agent 运行时）。
- **接入即注册**：任何接入的 Agent、人员、信息源在系统里都有身份记录，包括能产生什么事件、能接收什么动作、归属于哪个项目。

**可拓展方向**（不在 MVP 范围内）：

- 接入外部 Agents 和人员（身份与权限问题）
- 接入更多信息源（Jira、Figma、部署日志等）
- 向外部系统写出（如分配 GitHub Issues assignee — 需判断是否属于流程管理边界内）

### 3.2 流程引擎

**职责**：维护流程实例的状态和节点生命周期，定义流程的抽象结构。

**核心概念**：

- **流程模板**：定义节点和节点间的依赖关系，可实例化为具体的流程实例。
- **流程实例**：一条具体的流程执行，有自己的状态、归属项目、负责的 Agent。
- **节点**：流程中的一个环节，内部有 Agent 规划的子步骤、完成条件、评审材料。

**两种流程结构**：

- **开发流程（DAG 实例）**：单实例、长生命周期、强顺序依赖。大节点为 `Idea → Design → Dev & Test → Release`，节点间可有并行空间（如前后端并行开发），由 DAG 依赖关系表达。
- **维护流程（任务池）**：多实例、短生命周期、弱顺序依赖。bug fix、依赖升级、文档补全等条目在池中独立流转，各条目之间基本无依赖。

**并发处理**：项目可同时拥有多个活跃的流程实例（一个进行中的开发实例 + 若干维护条目）。Agent 不在流程之间切换，而是分别在各自结构内推进。项目层面有一个轻量协调者负责全局视图。

**节点状态流转**：

- **批准 → 推进到下一节点**：当前节点标记完成，Agent 沉淀内容，初始化下一节点。
- **驳回 → 根据评审意见重新构建节点内动作**：Agent 解析评审意见，调整节点内子步骤，不是简单回退到节点开始，而是带反馈重新规划。

### 3.3 决策机构

**职责**：在流程引擎产生的节点上做判断、推动或暂停流转。

由两个部分组成：

**Agents**，承担三个职责：

1. **自身/Team 动作决策**：判断节点边界条件是否满足、下一步应该做什么、是否需要协作。
2. **评审提交**：节点需要人工介入时，归纳该节点发生了什么、产出了什么、人需要判断的核心问题。
3. **知识沉淀**：无论推进还是驳回，将本次流转内容结构化写入知识库。

Agent 还具有**主动分析能力**：定期对项目数据做横向分析（如 PR 合并率、驳回最多的节点、反复出现的维护条目类型），这类分析本身也是知识沉淀的一部分。

**人工评审队列**：接收 Agent 提交的评审材料，人进行批准或驳回，并在驳回时提供意见。

**决策分级**：

- **低风险决策**（Agent 可自主执行）：超期未 review 的 PR 提醒、无 assignee 的 issue 推送、完成后触发下一节点
- **高风险决策**（Agent 提议，人批准）：项目模式切换（开发→维护）、优先级调整、成员分配、知识提升为共享

### 3.4 知识库

**职责**：接收和存储 Agent 沉淀的知识，提供检索能力。

**统一构建，三层分类**：

```
知识库（统一底层）
├── 项目知识库（自动，高频，项目私有）
│   来源：项目 Agent 在流程流转中自然沉淀
│   内容：决策记录、评审历史、项目特有的坑、节点推进/驳回记录
│
├── 流程共享知识（半自动，中频，职能维度）
│   来源：职能 Agent 在跨项目处理同类节点时积累
│   内容：流程执行经验（什么样的 Design 容易被驳回、Release 常见的阻塞原因）
│
└── 技术共享知识库（人工确认，低频，全局共享）
│   来源：Agent 提议 + 人确认提升
│   内容：前端规范、部署流程、通用技术判断
```

**知识流动方向**：单向向上提升，不会反向污染。项目知识不会自动变成技术共享知识，必须经过判断和确认。

**检索入口**：

- **项目入口**：默认只看本项目知识，可选拉入相关共享知识
- **全局入口**：跨项目检索，用于职能 Agent 和新成员 onboarding

**知识归属**天然由产生者决定：项目 Agent 产生的归属项目，职能 Agent 产生的归属流程共享，只有技术共享知识库需要主动决策。

## 项目数据模型

项目是系统中的核心实体，作为独立抽象存在，可实例化到流程中进行操作。

```
项目
├── 元信息：名称、状态、仓库地址、飞书群引用
├── 成员：归属于本项目的人员
├── Agents：归属于本项目的项目 Agent
├── 流程实例
│   ├── 开发流程实例（0~N 个，DAG 结构）
│   └── 维护条目（0~N 个，任务池结构）
└── 知识库入口：指向统一知识库中归属本项目的知识
```

## Agent 的两种分类

### 项目 Agent

归属于某个项目，负责该项目内的流程推进和知识沉淀。生命周期与项目绑定。自然沉淀**项目知识库**。

### 职能 Agent

跨项目存在，关注某个能力维度（前端、后端、文档、测试等）。在多个项目的同类节点中积累经验。自然沉淀**流程共享知识**。

两类 Agent 的关系是正交的：一个项目的 Design Review 节点，可能同时涉及项目 Agent（了解本项目的上下文）和前端职能 Agent（了解前端 Review 的通用标准）。 -->
