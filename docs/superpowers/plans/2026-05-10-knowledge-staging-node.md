# Knowledge Staging Node Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [x]`) syntax for tracking.

**Goal:** Allow `POST /projects/:id/entries` to omit `nodeId` by automatically anchoring the entry to a per-project staging node, while preserving the invariant that every `KnowledgeEntry` has a concrete node anchor.

**Architecture:** Add `NodeRole` to distinguish regular, project root, and staging root nodes without multiple booleans. Graph owns staging node creation and protection; Knowledge uses it when `nodeId` is absent and continues to validate explicit node anchors.

**Tech Stack:** NestJS, Prisma 7, PostgreSQL, Vitest, TypeScript, pnpm.

Add:

```ts
async findOrCreateStagingNode(projectId: string): Promise<Node> {
  const existing = await this.prisma.node.findFirst({
    where: { projectId, role: NodeRole.staging_root },
  })
  if (existing) return existing

  const root = await this.findProjectRoot(projectId)
  if (!root) throw new NotFoundException(`Project root not found for projectId=${projectId}`)

  return this.prisma.$transaction(async (tx) => {
    const staging = await tx.node.create({
      data: {
        projectId,
        role: NodeRole.staging_root,
        type: NodeType.staging,
        title: '[Staging Area]',
        createdBy: CreatedBy.human,
      },
    })
    await tx.edge.create({
      data: {
        projectId,
        fromId: root.id,
        toId: staging.id,
        type: EdgeType.composition,
        createdBy: CreatedBy.human,
      },
    })
    return staging
  })
}
```

## Self-Review

- Spec coverage: This plan covers project-level staging node, optional `nodeId`, resolved non-null `KnowledgeEntry.nodeId`, reanchor reuse, staging protection, and tests.
- Scope: It intentionally does not introduce a `Graph` table, per-subgraph staging graphs, event clustering, or entry merge/split behavior.
- Type consistency: `NodeRole` uses `regular`, `project_root`, and `staging_root`; `NodeType.staging` is only for staging nodes; explicit business anchors remain `role=regular` and `type=scaffold | growth`.
