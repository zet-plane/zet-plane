# 社团项目流程管理与知识沉淀系统 — 起点文档

> 本文档整理自一次完整的设计讨论，记录已形成共识的设计决策、系统结构、核心概念定义，以及需要在后续开发中通过实验验证的待定项。

---

## 一、项目背景与核心需求

### 要解决的现实问题

社团在项目开发中长期面临一些困难：

1. **成员流动导致知识断层**
   - 老成员积累的隐性知识（技术决策原因、模块历史坑点、被否决的方案）在毕业离开时几乎完全丢失，新成员需要从零摸索。
   - 进行到一半的开发任务由于成员个人因素中止，接替人无法全面了解开发情况、只能另起炉灶
2. **开发流程混乱**：任务分配靠口头和群内交流，进度不透明、工作内容难确定，日常事务和开发混在一起。



### Necessity

1. 渐进式知识沉淀

   - 相较于传统知识沉淀，可能是以飞书文档或博客的形式将某一次重要功能点开发沉淀。问题是其以结果为导向，举个例子：

     一个 commit，我们看到的是最终的 Comment 与 最终代码变更，后来者查看commit时也能够用Claude Code/Codex 理解八九成的代码含意，但无法获知细微的代码行变更上下文（可能由外部因素导致或系统Bug的边界处理）

   - 渐进式知识沉淀面向整个开发过程，会将沉淀意识在开发时就注入给系统的使用者，从而实现全流程的完整知识沉淀，披露变化、逐步完善，实现真实跟踪而非依据结果的倒推

2. Scaffold Graph

   - 任务分配靠口头和群内交流，进度不透明、工作内容难确定，日常事务和开发混在一起。我们可以通过节点化约束来实现流程确定，但是无法合理地进行流程控制、会导致很多的 **流程中磨损**。
   - 但是Scaffold Graph的意义是提供知识沉淀所面向的“过程”，其实质是一种引导而非约束。针对一个大的Graph，其形态是在Scaffold引导下为更好地实现需求而自由生长的。

3. 无感整理与上传

   - 我们可以通过人来编写文档、组织整理上传内容，但是人本质是懒惰的，同时整理上传也很有可能打断人的工作状态
   - 通过Agent(AI)来自动化、无感地整理过程中信息并上传，不仅是对项目成员的减负，同时也能更好、更稳定地实现流程引导和知识沉淀。

4. 端侧无关性
   - 如果依赖于某一具体办公套件或者开发工具，会给用户带来很多限制、依赖于具体情景，对不同项目不具有普适性。
   - 平面不依赖端侧实现，可以通过Adaptor/Integration接入、输出数据与结果

## 二、系统定位与核心边界



### 数据源

1. GitHub Issue, Commit, Review, Insight, etc.
2. 飞书群、QQ群
3. 飞书云文档、多维表格
4. bot、埋点、基建平台（Aliyun、TencentCloud、Grafana、Lens…）等监控信息
5. 用户输入





### 定位

**流程管理和知识沉淀系统**，管理"项目的流程"，不管"项目的内容"。

### 核心边界

- Agent 能做：读取 GitHub 数据、分析 PR 内容、生成摘要、更新流程状态、推送通知、写知识库、统计分析
- Agent 不做：写代码、改配置、提 PR、合并分支、修改项目内部任何内容
- 所有对项目本身的修改，由真人成员自己完成

### 与工作流（Coze/GitHub Actions）的分工

- **工作流**负责：已知流程的自动化（PR 合并通知飞书、定时生成报告模板、新成员自动创建页面）；已发生事件的结构化记录和传递
- **本系统（Agent）**负责：从记录中提炼理解（综合多个 PR、讨论、issue 给出连贯解释）；在开放情境下做判断（评估节点完成条件、归纳评审材料）；主动发现问题（识别无人维护的模块、反复出现的 bug 类型）

两者不是替代关系，是流水线上下游：**工作流负责捕获和触发，Agent 负责理解和回答。**

## 三、系统组成部分

系统由四个部分构成，形成以下依赖关系：

```
接入平面（读/写外部世界）
    ↓
流程引擎（维护实例状态和节点生命周期）
    ↓
决策机构
    ├── Agents（动作决策 + 评审提交 + 知识沉淀）
    └── 人工评审（批准/驳回，实际执行修改）
    ↓
知识库（接收 Agent 沉淀的内容）
```

### 3.1 接入平面

**职责**：连接外部世界与系统内部，双向通道。

- **读入方向**（事件标准化）：把飞书消息、GitHub webhook、定时任务等各种信息源的原始数据转化为系统内部的标准事件格式，流程引擎只消费标准事件，不关心数据来自哪里。
- **写出方向**（动作路由）：把系统内部的决策（Agent 建议、人的批准）路由到外部系统，如飞书通知、GitHub 打标签等。

**设计原则**：

- **标准事件格式**：新增信息源只需写一个新的 adapter，不需要改流程引擎（参考 Paperclip 的 adapter 模式，但应用于信息源而非 Agent 运行时）。
- **接入即注册**：任何接入的 Agent、人员、信息源在系统里都有身份记录，包括能产生什么事件、能接收什么动作、归属于哪个项目。

**可拓展方向**（不在 MVP 范围内）：

- 接入外部 Agents 和人员（身份与权限问题）
- 接入更多信息源（Jira、Figma、部署日志等）
- 向外部系统写出（如分配 GitHub Issues assignee — 需判断是否属于流程管理边界内）

### 3.2 流程引擎

**职责**：维护流程实例的状态和节点生命周期，定义流程的抽象结构。

**核心概念**：

- **流程模板**：定义节点和节点间的依赖关系，可实例化为具体的流程实例。
- **流程实例**：一条具体的流程执行，有自己的状态、归属项目、负责的 Agent。
- **节点**：流程中的一个环节，内部有 Agent 规划的子步骤、完成条件、评审材料。

**两种流程结构**：

- **开发流程（DAG 实例）**：单实例、长生命周期、强顺序依赖。大节点为 `Idea → Design → Dev & Test → Release`，节点间可有并行空间（如前后端并行开发），由 DAG 依赖关系表达。
- **维护流程（任务池）**：多实例、短生命周期、弱顺序依赖。bug fix、依赖升级、文档补全等条目在池中独立流转，各条目之间基本无依赖。

**并发处理**：项目可同时拥有多个活跃的流程实例（一个进行中的开发实例 + 若干维护条目）。Agent 不在流程之间切换，而是分别在各自结构内推进。项目层面有一个轻量协调者负责全局视图。

**节点状态流转**：

- **批准 → 推进到下一节点**：当前节点标记完成，Agent 沉淀内容，初始化下一节点。
- **驳回 → 根据评审意见重新构建节点内动作**：Agent 解析评审意见，调整节点内子步骤，不是简单回退到节点开始，而是带反馈重新规划。

### 3.3 决策机构

**职责**：在流程引擎产生的节点上做判断、推动或暂停流转。

由两个部分组成：

**Agents**，承担三个职责：

1. **自身/Team 动作决策**：判断节点边界条件是否满足、下一步应该做什么、是否需要协作。
2. **评审提交**：节点需要人工介入时，归纳该节点发生了什么、产出了什么、人需要判断的核心问题。
3. **知识沉淀**：无论推进还是驳回，将本次流转内容结构化写入知识库。

Agent 还具有**主动分析能力**：定期对项目数据做横向分析（如 PR 合并率、驳回最多的节点、反复出现的维护条目类型），这类分析本身也是知识沉淀的一部分。

**人工评审队列**：接收 Agent 提交的评审材料，人进行批准或驳回，并在驳回时提供意见。

**决策分级**：

- **低风险决策**（Agent 可自主执行）：超期未 review 的 PR 提醒、无 assignee 的 issue 推送、完成后触发下一节点
- **高风险决策**（Agent 提议，人批准）：项目模式切换（开发→维护）、优先级调整、成员分配、知识提升为共享

### 3.4 知识库

**职责**：接收和存储 Agent 沉淀的知识，提供检索能力。

**统一构建，三层分类**：

```
知识库（统一底层）
├── 项目知识库（自动，高频，项目私有）
│   来源：项目 Agent 在流程流转中自然沉淀
│   内容：决策记录、评审历史、项目特有的坑、节点推进/驳回记录
│
├── 流程共享知识（半自动，中频，职能维度）
│   来源：职能 Agent 在跨项目处理同类节点时积累
│   内容：流程执行经验（什么样的 Design 容易被驳回、Release 常见的阻塞原因）
│
└── 技术共享知识库（人工确认，低频，全局共享）
│   来源：Agent 提议 + 人确认提升
│   内容：前端规范、部署流程、通用技术判断
```

**知识流动方向**：单向向上提升，不会反向污染。项目知识不会自动变成技术共享知识，必须经过判断和确认。

**检索入口**：

- **项目入口**：默认只看本项目知识，可选拉入相关共享知识
- **全局入口**：跨项目检索，用于职能 Agent 和新成员 onboarding

**知识归属**天然由产生者决定：项目 Agent 产生的归属项目，职能 Agent 产生的归属流程共享，只有技术共享知识库需要主动决策。

## 四、"项目"的数据模型

项目是系统中的核心实体，作为独立抽象存在，可实例化到流程中进行操作。

```
项目
├── 元信息：名称、状态、仓库地址、飞书群引用
├── 成员：归属于本项目的人员
├── Agents：归属于本项目的项目 Agent
├── 流程实例
│   ├── 开发流程实例（0~N 个，DAG 结构）
│   └── 维护条目（0~N 个，任务池结构）
└── 知识库入口：指向统一知识库中归属本项目的知识
```

## 五、Agent 的两种分类

### 项目 Agent

归属于某个项目，负责该项目内的流程推进和知识沉淀。生命周期与项目绑定。自然沉淀**项目知识库**。

### 职能 Agent

跨项目存在，关注某个能力维度（前端、后端、文档、测试等）。在多个项目的同类节点中积累经验。自然沉淀**流程共享知识**。

两类 Agent 的关系是正交的：一个项目的 Design Review 节点，可能同时涉及项目 Agent（了解本项目的上下文）和前端职能 Agent（了解前端 Review 的通用标准）。

## 六、与既有工具的关系

| 工具 | 角色 | 系统如何与之交互 |
|---|---|---|
| GitHub | 代码仓库、PR、Issues | 接入平面读取事件，作为流程节点的外部数据源 |
| 飞书 | 沟通、通知、文档 | 接入平面读取消息/事件，写出方向推送通知 |
| GitHub Actions | CI/CD 自动化 | 继续使用，处理确定性工作流（构建、部署、lint） |
| Coze 等工作流 | 固定流程自动化 | 可选使用，处理结构化事务（新成员 onboarding 流程等） |
| 本系统 | 流程管理 + 知识沉淀 | 不替代以上工具，而是在它们之上提供理解和判断 |

## 七、相关项目参考

在设计过程中对比分析了以下项目：

| 项目 | 可借鉴的点 | 不适用的点 |
|---|---|---|
| **Multica** | Agent 作为 workspace first-class member 的交互模式；四种触发方式（assign/mention/chat/autopilot） | 纯 coding agent 管理工具，无知识沉淀和流程抽象 |
| **Paperclip** | Adapter 模式（接入任意 runtime）；Heartbeat 协议（定时唤醒 + session state 恢复）；Goal hierarchy（任务携带完整目标链条）；Atomic task checkout | 完全自治公司隐喻，不适合人机协作的社团场景 |
| **OpenAgents** | 共享运行环境思路（agent 共享同一个执行上下文） | 偏运行时共享，无流程管理 |
| **APM** | Context Handoff 机制（context 满了做结构化转移而非丢弃） | 纯 prompt 框架，不是持续运行的服务 |
| **CrewAI** | Crew 抽象（角色 + 任务 + 协作模式）；共享记忆系统 | 偏 SDK/框架，需要写代码配置 |

## 八、待实验清单

以下决策在设计阶段无法得出最优解，需要在开发中通过实际 testing 和实验分析后选择：

### 8.1 知识沉淀相关

| 编号 | 问题 | 可选方案 | 影响范围 |
|---|---|---|---|
| E-01 | 职能 Agent 沉淀流程共享知识的时机 | A: 每次处理完一个节点就沉淀 / B: 积累一定量后归纳再沉淀 | 流程共享知识库的可用性与碎片化程度 |
| E-02 | 知识提升为技术共享的触发机制 | A: Agent 定期扫描提议 / B: 职能 Agent 在处理中实时识别 / C: 人主动标记 | 技术共享知识库的质量和覆盖面 |
| E-03 | 项目知识库的自动化沉淀能从 PR 描述和 issue 中提取多少有效信息 | 取决于现有 PR/issue 的书写质量，需在真实数据上测试 | 知识库输入质量的上限 |

### 8.2 流程引擎相关

| 编号 | 问题 | 可选方案 | 影响范围 |
|---|---|---|---|
| E-04 | 评审驳回后，Agent 解析自然语言意见并重构节点内动作的质量 | 需要在真实评审场景中测试 Agent 的转化准确率 | 驳回路径是否真正可用 |
| E-05 | 开发流程 DAG 的粒度——大节点内部需要多细的子步骤 | A: Agent 自动细分 / B: 人定义模板 + Agent 填充 / C: 完全由 Agent 规划 | 流程的灵活性 vs 可预测性 |

### 8.3 Agent 决策相关

| 编号 | 问题 | 可选方案 | 影响范围 |
|---|---|---|---|
| E-06 | 低风险/高风险决策的边界在实践中是否清晰 | 需要积累实际案例后回顾分类是否合理 | 成员对系统的信任度 |
| E-07 | Agent 主动分析的频率和维度 | A: 定时全量分析 / B: 事件驱动增量分析 / C: 混合 | 分析价值 vs 计算成本 |

### 8.4 接入平面相关

| 编号 | 问题 | 可选方案 | 影响范围 |
|---|---|---|---|
| E-08 | 标准事件格式的具体 schema 设计 | 需在接入第一批信息源后根据实际数据迭代 | 整个系统的数据流 |
| E-09 | 接入平面的写出边界——哪些外部操作属于流程管理，哪些属于项目内容 | 如 GitHub Issues assignee 分配：流程管理 or 项目内容？需实际使用后判断 | 系统边界的清晰度 |

## 九、建议切入路径

**第一步：流程引擎 + 项目 Agent，端到端验证一个开发流程实例**

选一个真实的功能需求，用手动方式（飞书消息或 GitHub issue）触发，走完 `Idea → Design → Dev & Test → Release` 四个大节点。每个节点：

- Agent 检测边界条件
- Agent 归纳评审材料提交给人
- 人批准或驳回
- Agent 沉淀本节点知识

验证目标：这条链路跑通后，新成员能否从知识库里检索到有价值的项目历史。

**第二步：基于第一步的结果，决定下一个扩展方向**

- 如果知识沉淀效果好 → 扩展更多流程类型（维护流程任务池）
- 如果接入是瓶颈 → 做接入平面的标准化
- 如果 Agent 决策质量不够 → 改进 Agent 的 prompt 设计和上下文注入
