## 项目背景与核心信息

### 要解决的现实问题

社团在项目开发中长期面临一些困难：

1. **成员流动导致知识断层**
   - 老成员积累的隐性知识（技术决策原因、模块历史坑点、被否决的方案）在毕业离开时几乎完全丢失，新成员需要从零摸索。
   - 进行到一半的开发任务由于成员个人因素中止，接替人无法全面了解开发情况、只能另起炉灶
2. **开发流程混乱**：任务分配靠口头和群内交流，进度不透明、工作内容难确定，日常事务和开发混在一起。



### 核心功能方向

1. 渐进式知识沉淀

   - 相较于传统知识沉淀，可能是以飞书文档或博客的形式将某一次重要功能点开发沉淀。问题是其以结果为导向，举个例子：

     一个 commit，我们看到的是最终的 Comment 与 最终代码变更，后来者查看commit时也能够用Claude Code/Codex 理解八九成的代码含意，但无法获知细微的代码行变更上下文（可能由外部因素导致或系统Bug的边界处理）

   - 渐进式知识沉淀面向整个开发过程，会将沉淀意识在开发时就注入给系统的使用者，从而实现全流程的完整知识沉淀，披露变化、逐步完善，实现真实跟踪而非依据结果的倒推

2. Scaffold Graph（流程引导图）

   Scaffold Graph 是系统对开发过程的结构化描述，以节点和依赖关系表达流程形态，作为知识沉淀的骨架。

   - 任务分配靠口头和群内交流，进度不透明、工作内容难确定，日常事务和开发混在一起。我们可以通过节点化约束来实现流程确定，但是无法合理地进行流程控制、会导致很多的 **流程中磨损**。
   - Scaffold Graph 的意义是提供知识沉淀所面向的「过程」，其实质是一种引导而非约束。针对一个大的 Graph，其形态是在 Scaffold 引导下为更好地实现需求而自由生长的。

3. 无感整理与上传

   - 我们可以通过人来编写文档、组织整理上传内容，但是人本质是懒惰的，同时整理上传也很有可能打断人的工作状态
   - 通过Agent(AI)来自动化、无感地整理过程中信息并上传，不仅是对项目成员的减负，同时也能更好、更稳定地实现流程引导和知识沉淀。

4. 端侧无关性
   - 如果依赖于某一具体办公套件或者开发工具，会给用户带来很多限制、依赖于具体情景，对不同项目不具有普适性。
   - 平面不依赖端侧实现，可以通过Adaptor/Integration接入、输出数据与结果



### 系统定位

现有工具已经很好地覆盖了「已发生的事」：GitHub 记录代码变更，飞书沉淀讨论，CI/CD 自动化确定性流程。但它们共同留下了一个空白——**过程中的理解**。

一条 PR 合并了，没有人记录为什么这样设计而不是另一种方式；一个节点卡了两周，没有系统知道卡在哪里、为什么卡；一位老成员毕业，带走的不是代码，而是判断力。

**本系统（Zet Plane，以下简称「平面」）定位于这个空白**：它不是代码仓库、不是项目管理看板、不是 IM 工具，而是运行在这些工具之上的**流程引导层与知识沉淀层**。它的目标用户是**成员流动频繁的社团或小型开发团队**，核心价值是：让每一次流程流转都留下可被后来者理解的痕迹，让项目进度更可见、可跟随，让新成员能从知识库中复现团队的判断力而不必从零摸索。



### 核心能力边界

边界不是规则列表，而是四条可推导的判断原则。当遇到模糊情况时（如「给 Issue 分配 assignee 算不算越界」），用这四条原则做判断：

**原则一：理解而非执行**

Agent 的动作范围止于「读取、分析、建议、记录」。凡是会在**项目内容**上留下痕迹的操作——写代码、改配置、提 PR、合并分支、修改仓库内任何文件——均超出边界。系统只在**自身状态**上写入（流程记录、知识库条目、评审材料）。

**原则二：引导而非约束**

Scaffold Graph 提供的是过程形态的引导，不是执行路径的锁定。系统不应成为开发流程的瓶颈——节点可以自由生长，Agent 的职责是让生长留下记录，而非控制生长的方向或速度。

**原则三：沉淀面向人，而非面向 Agent**

知识沉淀的目标是让**没有任何上下文的后来者**能独立理解历史，而非仅服务于当前 Agent 的上下文维持。每次沉淀应当能回答：「一个新成员在没有 Agent 的情况下，能从这条记录中复现当时的判断吗？」

**原则四：适配端侧，而非依赖端侧**

系统的核心逻辑——流程状态、节点生命周期、知识库——不与任何具体的办公套件或开发平台绑定。飞书、GitHub、QQ 群等均通过 Adapter 接入，是可替换的外部端点，而非系统的内在假设。当某个端侧工具迁移、替换或下线时，系统的流程与知识沉淀不应受到结构性影响。

| 原则 | 核心判断 |
|---|---|
| 理解而非执行 | 是否在项目内容上留下痕迹？ |
| 引导而非约束 | 是否成为流程的瓶颈或锁死节点路径？ |
| 沉淀面向人 | 没有 Agent 的新成员能否从记录中独立复现判断？ |
| 适配而非依赖端侧 | 核心逻辑是否与具体工具解耦？ |



### 信息输入范围

信息源的选择原则：**凡能反映开发过程、决策轨迹、运行状态的来源优先纳入**；信息源以事件驱动为主，静态内容不做持续监控，但 Agent 在分析过程中可按需读取历史快照（如特定 commit 的代码状态）。

| 类别 | 具体来源 | 价值 |
|---|---|---|
| **代码与协作** | GitHub Issue、Commit、PR、Review、Insight | 开发过程的主要载体，决策与变更的直接记录 |
| **沟通与文档** | 飞书群、QQ 群、飞书云文档、多维表格 | 非正式讨论与决策的隐性记录，往往包含代码里看不到的上下文 |
| **系统与监控** | bot、埋点、Aliyun/TencentCloud、Grafana、Lens 等 | 运行状态与基建数据，辅助判断维护节点的触发条件 |
| **人工输入** | 用户直接输入 | 无法被自动捕获的上下文，如口头决策、临时说明 |

> 新增信息源通过接入平面的 Adapter 机制接入，不影响系统核心逻辑（见原则四）。
